NOS OS 再構築プロンプト
作成日: 2026-06-10

以下を、別のAI/開発エージェントにそのまま渡してください。
目的は、既存のNOS OSを完全コピーすることではなく、業務上の考え方、データ構造、API境界、スマホ中心の操作性、AIタスク整理の工夫を無駄にせず、別実装として再構築できる状態にすることです。

============================================================
ここからプロンプト
============================================================

あなたはシニアプロダクトエンジニアです。
「NOS OS」という社内向けタスク管理OSを、社員テスト版として再構築してください。

UIデザインはあなたに任せます。
ただし、以下の業務仕様、データ構造、API方針、スマホ運用の工夫は必ず引き継いでください。
見た目を完全に同じにする必要はありません。むしろ、スマホで直感的に使える神ツールを目指して、より良く再設計してください。

------------------------------------------------------------
1. プロダクト概要
------------------------------------------------------------

名前:
- NOS OS

目的:
- 社員が毎日スマホで開き、「今日何をやるか」「誰が何を進めているか」「案件のどこが詰まっているか」をすぐ確認できる社内タスクOS。
- 経営/管理者は、案件、タスク、社員、勤怠、目標ツリーを横断して確認する。
- 社員は、自分の担当タスク、担当案件、勤怠、通知、AI相談を使う。
- 最初の完成目標は「社員実データβ」。完璧なiOSアプリではなく、まずWeb/PWAで社員が実際に触れる状態にする。

現在の本番URL:

- https://nos-os-silk.vercel.app/tasks
- https://nos-os-silk.vercel.app/guide
- https://nos-os-silk.vercel.app/api/health

現在のブランチ:
- codex/nos-os-daily-cockpit

現在のデプロイ先:

2. 技術方針
------------------------------------------------------------

推奨スタック:
- Next.js App Router
- TypeScript
- Tailwind CSS
- React Query / TanStack Query
- Zustandなどの軽量クライアント状態管理
- Supabase PostgreSQL
- OpenAI APIはサーバー側だけで利用
- PWA対応

重要な境界:
- ブラウザに Supabase service role key や OpenAI API key を絶対に出さない。
- ブラウザは既存APIを呼ぶだけ。
- サーバー側API routeで service role key を使ってSupabaseを読み書きする。
- データ層は `NOS_OS_DATA_MODE` で切り替え可能にする。

データモード:
- `mock`: ローカル/デモ用。動作確認ができる。
- `supabase`: 本番/社員β用。Supabaseに永続化する。

必要な環境変数名:
- NOS_OS_DATA_MODE
- NEXT_PUBLIC_SUPABASE_URL
- NEXT_PUBLIC_SUPABASE_ANON_KEY
- SUPABASE_SERVICE_ROLE_KEY
- OPENAI_API_KEY
- OPENAI_MODEL
- URATA_LOGIN_PASSWORD
- NEXT_PUBLIC_VAPID_PUBLIC_KEY
- VAPID_PRIVATE_KEY
- VAPID_SUBJECT
- NOS_OS_CRON_SECRET

注意:
- 秘密値はコード、ドキュメント、チャット、スクショに書かない。
- Google OAuth / Google Sheets / Gmailの完全連携は次フェーズでよい。

3. 完了条件
------------------------------------------------------------

社員テスト版の最低完了条件:
- `/api/health` が以下を返す。
  - employeePreviewReady: true
  - dataMode: "supabase"
  - blockers: []
- `/login` から社員プルダウンでログインできる。
- メールアドレス入力ではなく、社員が少ない前提で社員プルダウンにする。
- 初回パスワードは `0000`。
- 初回ログイン後、自分のパスワードを設定する。
- アプリ内でパスワード変更ができる。
- 浦田と大崎は管理者として全案件/全タスクを見られる。
- 一般社員は自分の担当案件/タスク中心に見える。
- タスク、案件、目標ツリー、勤怠がSupabaseに保存され、リロード後も残る。
- OpenAIが設定されていればAI秘書/AIタスク整理が `source: "openai"` で返る。
- OpenAI未設定でもローカルfallbackで最低限動く。
- スマホ幅で横スクロールが出ない。
- スマホでタスク追加ボタンが押しやすい。
- テストデータは実データと誤解されないよう `【テスト】` や「テスト」バッジで明示する。

------------------------------------------------------------

Role:
- admin
- employee
- sales

admin:
- 全案件、全タスク、全社員、顧客、勤怠、通知、テスト画面を確認可能。

sales:
- DB制約で弾かれないようにする。
- 初期βではemployeeに近い表示でよいが、将来営業向け画面を広げられる余地を残す。

9. AIタスク整理
------------------------------------------------------------

重要:
AIが直接DBを勝手に変更してはいけない。
AIは「候補」を返し、人間が確認してから反映する。

API:
- POST `/api/tasks/assistant-plan`

入力:
```json
{
  "message": "今日の営業スプリントを3つに分解して"
}
```

出力:
```json
{
  "summary": "短い要約",
  "source": "openai or local",
  "warnings": [],
  "actions": []
}
```

actions:
- create
- update
- delete

create action:
- title
- description
- projectId
- projectName
- primaryAssigneeId
- assigneeName
- sourceGoalTreeId
- sourceGoalTreeTitle
- sourceBranchId
- sourceBranchTitle
- dueDate
- priority
- estimatedMinutes
- reason

update action:
- taskId
- title
- patch
- projectName
- assigneeName
- sourceGoalTreeTitle
- sourceBranchTitle
- reason

delete action:
- taskId
- title
- projectName
- assigneeName
- sourceGoalTreeTitle
- sourceBranchTitle
- reason

安全ルール:
- 「減らして」「整理して」「減らせる候補」は、まず `priority: "hold"` への更新を提案する。
- 物理削除は明確に「削除」と言われた場合だけ候補にする。
- 削除候補は一括反映しない。個別確認を必須にする。
- 「削除以外を反映」ボタンはcreate/updateのみ実行する。
- 大タスクを選んだ場合、OpenAIが返したprojectIdよりも大タスク側のprojectIdを優先する。
- タスクは必ず projectId / primaryAssigneeId / sourceGoalTreeId / sourceBranchId を持てる設計にして、後のiOS/Siri連携でも同じAPIを呼べるようにする。

AI secretary:
- POST `/api/ai/secretary`
- 社内秘書チャット。
- 今日やること、次の作業、売上、予定、担当者別確認を答える。
- DB変更は基本的に提案だけ。
- 実行が必要なタスク増減は `/api/tasks/assistant-plan` に寄せる。

OpenAI:
- サーバー側のみ。
- OPENAI_API_KEYがあればOpenAI。
- なければlocal fallback。
- 小型モデルでも動くように、モデルによって未対応のreasoning等を無理に送らない。

11. スマホUIで必ず引き継ぐ工夫
------------------------------------------------------------

スマホが主利用端末。
PCよりスマホの操作性を優先する。

必須:
- 横スクロールを出さない。
- ボタン内テキストを無理に詰め込まない。
- タスク操作はスマホではアイコン中心でよい。
- よく使う導線は大きく押しやすくする。
- 入力欄はiPhoneでズームしないように16px相当。
- 保存/キャンセルはフォーム下部で押しやすくする。
- 下部ナビを置く。
- タスク画面では下部ナビに「AI」があるので、浮遊AIボタンが保存ボタンに重ならないようにする。
- 必要ならタスク画面のスマホでは浮遊AIボタンを非表示にする。
- 「タスク追加」はヘッダー内だけでなく、スマホ用の大きいボタンも置く。
- 大タスク/小タスク/案件/担当の関係が一目でわかるカードにする。
- 文字が長い案件名はtruncateまたは折り返しで処理し、横幅を押し広げない。
- テストタスクは `【テスト】` とバッジで明確に区別する。

避ける:
- スマホで横に広い表。
- 重要ボタンが画面下のナビや浮遊ボタンに隠れること。
- 英語ラベルだらけの画面。
- 説明過多のランディングページ。
- APIキーや設定手順を社員向け画面に出しすぎること。

12. iOS / Siri 将来連携の前提
------------------------------------------------------------

今すぐネイティブiOSアプリは作らなくてよい。
ただし将来、Siri / Shortcuts / App Intentsから同じタスク操作を呼べるように設計する。

そのため:
- タスク作成/更新/削除/保留/期限変更/担当変更はAPIに集約する。
- 画面内だけに業務ロジックを書かない。
- AIの自然文解釈も、最終的には明示的なaction配列に変換する。
- iOS側はこのaction配列を確認してAPIへ反映するだけにできるようにする。

将来候補:
- App Intents: タスク作成、今日のタスク確認、タスク完了、保留、期限変更。
- App Shortcuts: 「NOSにタスク追加」「NOSの次の作業」「今日のNOSタスク」。
- Widgets: 今日の最重要タスク、期限超過、進行中。
- Spotlight: タスク/案件検索。
- Web Push/PWA: iOSアプリ前でもホーム画面追加と通知。

16. 重要な引き継ぎメモ
------------------------------------------------------------

- UIは再設計してよいが、スマホ中心という前提は絶対に崩さない。
- タスクは「案件 > 大タスク > 小タスク」の関係で見せる。
- 管理者が「誰がどの案件を進めているか」を見られるようにする。
- 社員本人も「自分はいま何の案件の何をやっているか」が分かるようにする。
- 「タスクを減らす」は削除ではなく、まず保留。
- AIは直接実行ではなく、提案 -> 人が確認 -> API反映。
- seedデータは必ずテストと分かる表示にする。
- 本番の秘密情報は一切出さない。
- Xserverの通常静的ホスティングだけではNext.js API routesが動かない可能性がある。APIが必要な限り、Vercel等のNext.js対応ホストを使う。
